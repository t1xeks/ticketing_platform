# Подробное объяснение проекта `ticketing_platform`

## 1) Что это за проект простыми словами

Это веб-приложение "HelpDesk / Ticketing System", где:
- обычный сотрудник (employee) создает заявки о проблемах (например, не работает принтер);
- поддержка (support) видит все заявки и меняет их статус;
- данные хранятся в базе PostgreSQL;
- есть API (интерфейс для программ), которое документировано в Swagger.

---

## 2) Что я изменил в архитектуре

Раньше почти вся логика была в одном файле `app.py`.  
Сейчас проект разделен на слои (это считается "нормальной структурой"):

- `models` - только модели базы данных (таблицы).
- `services` - бизнес-логика (правила приложения).
- `web` - контроллеры/роуты для HTML-страниц.
- `api` - контроллеры/роуты для JSON API.
- `exceptions` - единая обработка ошибок.
- `static/js` - клиентская логика на фронте (JavaScript).

Главная идея: **контроллеры стали "тонкими"**, а вся важная логика вынесена в сервисы.

---

## 3) Новая структура файлов

### Точка входа
- `app.py`  
  Запускает приложение через фабрику `create_app()`.

### Пакет приложения
- `ticketing_app/__init__.py`  
  Создает Flask-приложение, подключает БД, login, csrf, swagger, блюпринты, обработчики ошибок, создает таблицы и начального админа.

- `ticketing_app/config.py`  
  Конфиг приложения (подключение к БД, `SECRET_KEY` и т.д.).

- `ticketing_app/extensions.py`  
  Инициализация расширений Flask:
  - `db` (SQLAlchemy),
  - `login_manager` (авторизация),
  - `csrf` (защита форм),
  - `swagger` (документация API).

### Модели (database layer)
- `ticketing_app/models/employee.py` - таблица сотрудников.
- `ticketing_app/models/issue.py` - таблица заявок.
- `ticketing_app/models/support_staff.py` - таблица поддержки.
- `ticketing_app/models/user.py` - таблица пользователей для входа.
- `ticketing_app/models/__init__.py` - общий импорт моделей.

### Сервисный слой (business logic)
- `ticketing_app/services/auth_service.py`  
  Регистрация, вход, определение роли пользователя.

- `ticketing_app/services/issue_service.py`  
  Создание/чтение/обновление/удаление заявок, проверка доступа.

- `ticketing_app/services/account_service.py`  
  Обновление и удаление аккаунта, синхронизация email во всех нужных таблицах.

- `ticketing_app/services/__init__.py` - общий импорт сервисов.

### Контроллеры
- `ticketing_app/web/routes.py`  
  HTML-страницы (`/login`, `/issues`, `/account` и т.д.).

- `ticketing_app/api/routes.py`  
  JSON API (`/api/issues`, `/api/issues/<id>`).

### Ошибки
- `ticketing_app/exceptions.py`  
  Кастомные ошибки и глобальные error handlers.
  - Для API: возвращается JSON с ошибкой.
  - Для веба: показывается HTML-страница `error.html`.

### Формы
- `ticketing_app/forms.py`  
  Flask-WTF формы для логина, регистрации и заявок.

### Фронтенд
- `templates/client.html`  
  Новая клиентская страница.

- `static/js/client.js`  
  Фронтовая логика: fetch-запросы к API, отрисовка таблицы заявок, создание заявки через API.

### Шаблоны
- существующие HTML-шаблоны обновлены на новую маршрутизацию через блюпринт `web.*`.

## 4) Как работает поток запроса (очень важно для объяснения преподавателю)

Пример: сотрудник создает заявку.

1. Пользователь отправляет форму (или JS вызывает API).
2. Контроллер (`web/routes.py` или `api/routes.py`) принимает запрос.
3. Контроллер вызывает сервис (`IssueService.create_issue(...)`).
4. Сервис:
   - проверяет бизнес-правила;
   - читает/пишет данные через модели SQLAlchemy;
   - делает `commit`.
5. Контроллер возвращает ответ:
   - HTML-страница (web),
   - или JSON (api).

То есть контроллер почти не содержит логики - только "принял запрос -> вызвал сервис -> вернул ответ".

## 5) Какие ключевые функции использованы

## Flask
- `Flask(...)` - создание приложения.
- `Blueprint(...)` - разделение роутов по модулям.
- `render_template(...)` - рендер HTML.
- `request.get_json(...)` - чтение JSON-тела.
- `jsonify(...)` - JSON-ответ.
- `redirect(...)`, `url_for(...)`, `flash(...)`.

## Flask-Login
- `@login_required` - защита роутов.
- `login_user(...)` / `logout_user(...)`.
- `current_user`.
- `user_loader` - загрузка пользователя по id.

## SQLAlchemy
- `db.Model` - описание таблиц.
- `query.filter_by(...).first()/all()`.
- `db.session.add(...)`, `db.session.delete(...)`, `db.session.commit()`.

## Безопасность
- `generate_password_hash(...)`, `check_password_hash(...)`.
- CSRF защита форм через `Flask-WTF`.

## Swagger
- Подключен через `flasgger`.
- Документация доступна по адресу: `/api/docs/`.

## 6) API, которое теперь есть

После входа в систему:

- `GET /api/issues` - получить список заявок текущего пользователя (или всех, если support).
- `POST /api/issues` - создать заявку (только employee).
- `PATCH /api/issues/<issue_id>` - обновить заявку по правилам роли.
- `DELETE /api/issues/<issue_id>` - удалить заявку (только владелец).

Все эти методы описаны в Swagger UI.

---

## 7) Обработка исключений

Сделан централизованный обработчик ошибок:

- `ValidationError` - ошибки валидации (например, занятый email).
- `AuthorizationError` - нет прав доступа.
- `NotFoundError` - объект не найден.

Если запрос пришел на `/api/...`, пользователь получает JSON:
```json
{ "error": "текст ошибки" }
```

Если это обычная веб-страница, показывается шаблон `templates/error.html`.

## 8) Как запустить проект (пошагово для новичка)

## Вариант А: через Docker (самый простой)

1. Установить Docker Desktop.
2. Открыть терминал в папке проекта.
3. Выполнить:
   ```bash
   docker-compose up --build
   ```
4. Открыть в браузере:
   - приложение: `http://localhost:5000`
   - swagger: `http://localhost:5000/api/docs/`

---

## Вариант Б: без Docker (локально)

1. Установить Python 3.10+ и PostgreSQL.
2. Создать и активировать виртуальное окружение:
   ```bash
   python -m venv env
   # Windows:
   env\Scripts\activate
   ```
3. Установить зависимости:
   ```bash
   pip install -r requirements.txt
   ```
4. Создать БД и пользователя PostgreSQL как в `README.md`.
5. При необходимости задать переменную окружения `DATABASE_URL`.
6. Запустить:
   ```bash
   python app.py
   ```
7. Открыть:
   - `http://localhost:5000`
   - `http://localhost:5000/api/docs/`
